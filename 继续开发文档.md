### 1.2 拼盘界面重构（视觉摆盘）
**现状**：Plating 是纯文字 toggle 按钮，没有"摆盘"的视觉感。
**改进**：

- 右侧显示一个圆形"盘子"区域（CSS 圆形 + 内阴影模拟）
- 点击食材后，食材 emoji 出现在盘子内（绝对定位，随机散落）
- 可点击盘内食材移除，盘子外的是"备选区"
- 简单实现：不需要 drag-and-drop，click-to-add/remove 即可
- 实现复杂度：低～中

### 1.3 操作步骤撤销
**现状**：操作日志有删除按钮，但只能删末尾步骤（实际是按 step index 删）。
**改进**：

- 确认当前删除行为是否支持删任意步骤（需验证 UI）
- 加"撤销上一步"快捷按钮（Undo），提升操作流畅度
- 实现复杂度：极低

### 1.4 食材组合 Combo 提示
**现状**：玩家完全盲目操作，不知道什么搭配会好。
**改进**：

- 进入烹饪阶段后，助手给出一条"不靠谱建议"（AI 生成或预设模板）
- 例如："我觉得把棉花糖和芥末一起蒸会很好吃……不是吗？"
- 建议显示为可折叠的侧边提示条，不强制显示
- 可以在 roundStartAgent 增加 `assistantTip` 字段（选填，不影响现有结构）
- 实现复杂度：低

### 1.5 评分详情展开
**现状**：三项评分（食材/文化/创意）只显示数字，不解释为什么扣分。
**改进**：

- 结果页评分卡展开后显示每项的简短说明（1句话）
- 评分 JSON 中增加 `ingredientNote`、`culturalNote`、`creativityNote` 字段
- 修改 feedback prompt，要求 AI 在 score JSON 里附带每项简评
- 实现复杂度：低（改 prompt + 类型定义）

## 二、UI/UX 体验（中等优先级）

### 2.1 结果页动效
**现状**：评分和图片直接显示，没有任何过渡感。
**改进**：

- 菜品图片淡入（CSS `animate-fadeIn` 或 `transition-opacity`）
- 分数数字从 0 滚动到最终值（用 `requestAnimationFrame` 或 CSS counter）
- 评级文字延迟出现 + scale 弹出动画
- "神作"评级时加金色背景闪光（`animate-pulse` 配 amber 色）
- 实现复杂度：低～中（纯 CSS + 少量 JS）

### 2.2 烹饪操作台改版
**现状**：食材是一排平铺按钮，选中靠颜色区分，视觉层次不清晰。
**改进**：

- 食材改为卡片（emoji 大图 + 中文名），选中时边框高亮
- 操作日志改为气泡流风格（类似聊天记录，每步是一个卡片）
- 可选：操作完成时食材 emoji 短暂飞向操作日志区（CSS keyframe）
- 实现复杂度：中等（需重写 Inventory.tsx 布局）

### 2.3 加载动画优化
**现状**：loading 阶段只有一个跳动的 🍽️ 图标。
**改进**：

- 显示进度文案（"正在为你随机挑剔食客…" → "正在让助手拿错食材…" → "准备就绪！"）
- 文案随时间切换，给用户更好的等待体验
- 实现复杂度：极低（用 setInterval 切换文案）

### 2.4 移动端适配
**现状**：布局以桌面为主，移动端未经测试。
**改进**：

- 烹饪阶段改为竖向布局（食材列表在上，操作区在下）
- 按钮尺寸优化为 touch-friendly（`min-h-12 px-5`）
- 烹饪日志默认折叠，节省屏幕空间
- 实现复杂度：低（Tailwind 响应式断点调整）

## 三、内容扩充（中等优先级）

### 3.1 菜品难度标签
**现状**：菜品只有 `name` 和 `cuisine` 字段，没有难度区分。
**改进**：

- `dishes` 表增加 `difficulty` 字段（`easy` / `normal` / `hard`）
- 难度影响 AI 评分严格程度（hard 菜更容易被扣分）
- 主页可选"挑战模式"（只出现 hard 菜品）
- 实现复杂度：低（改 schema + seed + prompt 传参）

### 3.2 混乱事件扩充
**现状**：只有 4 种混乱事件，重复率较高。
**改进**：新增事件：

| 事件 | 效果 |
|------|------|
| 断电！ | 禁止 `fry`（明火替代） |
| 食材调包 | 随机把一个食材名替换成奇怪的东西（显示层） |
| 时间紧迫 | 步骤数限制为 3 步 |
| 神秘评委 | 本轮 AI 扮演更刁钻的食评人 |

- 实现复杂度：低（在 `roundStartAgent.ts` 的 CHAOS_EVENTS 数组追加）

### 3.3 分享结果功能
**现状**：游戏结果只能截图，没有原生分享。
**改进**：

- 结果页加"复制成绩卡"按钮，生成如下格式文本并复制到剪贴板：
  ```
  🍳 Chaos Chef
  食客：Sophie Beaumont（法国）
  菜品：北京炸酱面 × 棉花糖芥末冰淇淋
  得分：24/30 · 出色 🤩
  "我没想到你真的能把这个做得……勉强能吃"
  ```
- 实现复杂度：极低（`navigator.clipboard.writeText`）

## 四、技术改进

### 4.1 加载失败重试机制
**现状**：API 失败直接回开始页，Toast 提示但无重试入口。
**改进**：

- loading 阶段失败时不立即跳回，而是显示"加载失败"局部状态
- 提供"重试"按钮，重新触发 `/api/round/start` 请求
- 最多自动重试 1 次后再跳回开始页
- 实现复杂度：低

### 4.2 Mock 模式（本地开发）
**现状**：开发调试必须有有效 API key，迭代 UI 成本高。
**改进**：

- `NEXT_PUBLIC_MOCK_AI=true` 环境变量
- Mock 时跳过真实 AI 调用，返回预设假数据（固定食材/固定台词/固定评分）
- 方便纯前端 UI 迭代
- 实现复杂度：低（在 API route 里判断环境变量）

### 4.3 图片生成 Prompt 优化
**现状**：`buildImagePrompt` 生成图片描述后调用 DALL-E，但生成质量不稳定（食材混搭可能出奇怪结果）。
**改进**：

- 在 prompt 中强制加入"food photography style, studio lighting, clean plate"等摄影关键词
- 限制描述长度避免模型混乱
- 加 negative prompt（如果 API 支持）：`no text, no people`
- 实现复杂度：极低（改 `lib/prompts/imagePainter.ts`）
