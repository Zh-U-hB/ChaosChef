# Chaos Chef

## 游戏概念

**Chaos Chef** 是一款以 LLM Agent 为核心驱动的创意烹饪模拟游戏。区别于传统依赖预设规则和固定食谱的烹饪游戏，本作将 AI 大语言模型直接嵌入游戏逻辑层，充当一个"混乱裁判"——它理解烹饪文化、食材化学、地域饮食习惯，并据此对玩家的每一次临场发挥做出真实、有创意、且不可预测的判断。

游戏的核心设计哲学是**"创意替代"**：当正确的食材缺席时，玩家的即兴发挥是否能被 AI 识别为合理的烹饪创造？这个问题没有标准答案——LLM 会从全球饮食文化的视角去评估每一次替代的合理性，可能给出"意外的融合佳作"，也可能判定"文化灾难"或"令人惊叹的误打误撞"。

**Agent 原生设计**意味着游戏的核心判断、叙事生成、客人反应全部通过 AI Agent 管线实时运行，而非依赖预写的条件分支。每一局游戏都是独一无二的混乱体验。

### 世界观设定：不靠谱的助手

混乱的根源有一个具体的名字——你的厨房助手。

这位助手热情有余、能力不足，每次去采购食材时都会以某种神秘的方式拿错东西：可能是听错了名字、认错了标签，也可能只是觉得"这个看起来差不多"。他从不承认自己拿错了，还会用充满自信的语气把食材递给你，仿佛一切都在计划之中。

这个设定为游戏的核心矛盾提供了合理的叙事锚点——玩家不是在一个物资充足的理想厨房里工作，而是每次都要在助手"带回来的这堆东西"里想办法。食材的随机性不是系统 bug，而是世界观的一部分。

## 游戏说明

### 基本玩法

玩家经营一家食材供应永远"不太对劲"的混乱餐厅。来自世界各地的食客带着具体的菜品需求走进厨房——可能是一碗正宗的北京炸酱面、一份摩洛哥塔吉锅，或是一道经典的法式焦糖布丁。

然而，玩家的食材库存是随机且有限的：你可能只有菠萝、虾酱、意大利面和一块黄油。玩家必须在现有食材中做出选择，尽可能地还原客人点单的菜品，或者大胆创造一道"精神上接近"的替代品。

### AI 裁判机制

每道菜提交后，LLM Agent 会综合以下维度进行评估：

- **食材合理性**：替代食材在烹饪逻辑上是否说得通（如用番茄替代酸角）
- **文化契合度**：成品是否符合客人所在文化的饮食审美与禁忌
- **创意加分**：出人意料但令人信服的搭配会触发"灵感时刻"奖励
- **灾难判定**：严重的文化冒犯或食材禁忌会引发客人的强烈负面反应

AI 裁判的评语是自然语言生成的，每次都不同——客人可能会惊喜、困惑、感动，甚至拂袖而去。

### 客人系统

食客来自全球不同地区，各自携带独特的文化背景、口味偏好和饮食禁忌。同一道"替代菜品"可能让一位冒险的年轻背包客赞不绝口，却让一位坚守传统的老食客无法接受。了解客人背景、预判文化反应，是游戏的核心策略之一。

### 游戏节奏

每轮游戏在有限时间内接待多位客人，玩家需要在"尽量还原"与"大胆创新"之间做出取舍。随着游戏进程，客人要求的菜品愈发刁钻，食材库存愈发匮乏，但 AI 裁判对于绝妙创意的奖励也会更加慷慨。

## 技术方案

### 技术栈

| 层级 | 技术选型 | 说明 |
|------|----------|------|
| 前端框架 | Next.js 15 (App Router) | 全栈框架，兼顾页面渲染与 API 路由 |
| UI 框架 | React 19 | 组件化游戏界面 |
| 样式 | Tailwind CSS v4 | 快速构建响应式布局 |
| 客户端状态 | Zustand | 轻量游戏状态管理 |
| AI 模型（文本） | Claude claude-sonnet-4-6 | 点餐对话生成 + 食客反馈生成 |
| AI 模型（图像） | DALL-E 3 / fal.ai | 菜品成品图片生成 |
| AI SDK | Anthropic TypeScript SDK + OpenAI SDK | 分别对接文本与图像模型 |
| 数据库 | SQLite + Prisma | 存储食客数据与菜品名称 |
| 运行时 | Node.js | 服务端 Agent 逻辑 |

### 系统架构

```
浏览器 (Next.js 前端)
    │
    ├── GET /api/round/start      ─ 阶段一：回合启动
    ├── POST /api/dish/submit     ─ 阶段三：提交菜品
    │
    ▼
Next.js API Routes (服务端)
    │
    ├── 阶段一：回合启动 Agent
    │     ├── 从数据库随机取 → 食客数据 + 菜品名称
    │     └── 调用 Claude   → 点餐对话文本 + 助手带来的食材列表
    │
    └── 阶段三：结果生成 Agent（并行）
          ├── 调用 Claude   → 食客反馈文本（SSE 流式）
          └── 调用图像模型  → 菜品图片 URL
    │
    ▼
┌─────────────────────┬──────────────────┐
│  Anthropic Claude   │  图像生成模型     │
│  (文本反馈)          │  (菜品图片)       │
└─────────────────────┴──────────────────┘
    │
    ▼
数据库 (SQLite)
  ├── customers 表：食客姓名、来源地区、性格、饮食禁忌
  └── dishes 表：菜品名称、所属菜系
```

### Agent 设计

游戏核心分为三个阶段，由 Agent 驱动阶段一和阶段三：

#### 阶段一：回合启动 Agent

每轮开始时触发，完成数据拉取和内容初始化：

**Step 1 — 数据库查询（工具调用）**
- 随机取一条食客记录：姓名、来源地区、性格标签、饮食禁忌
- 随机取一个菜品名称：菜名、所属菜系

**Step 2 — LLM 生成（Claude）**
- 输入：食客数据 + 菜品名称
- 输出（结构化 JSON）：
  ```json
  {
    "orderDialogue": "服务员，我要一份...（食客说话的原文，带口音/性格特征）",
    "ingredients": ["食材A", "食材B", "食材C", "..."]
  }
  ```
  > `ingredients` 是助手"理解"后拿来的食材，刻意与正确食谱有偏差

---

#### 阶段二：玩家操作（纯客户端）

玩家对食材进行处理，操作过程实时记录到 Zustand 状态，不涉及服务端调用。

每个操作步骤记录为：
```json
{
  "step": 1,
  "action": "steam",          // 烹饪动作：steam/fry/bake/boil/raw...
  "ingredients": ["食材A", "食材B"],
  "duration": "5min"          // 可选参数
}
```

玩家完成所有操作并拼盘后，将操作序列打包提交。

---

#### 阶段三：结果生成 Agent（并行双模型）

接收玩家操作 JSON 后，**同时**触发两个生成任务：

**任务 A — 食客反馈（Claude，SSE 流式）**
- 输入：食客数据、原始点单菜品、玩家完整操作过程 JSON
- 输出（流式）：食客的自然语言反馈，包含满意/惊喜/愤怒等情绪，以及最终评分

**任务 B — 菜品图片（图像模型）**
- 输入：根据玩家操作过程自动拼装 image prompt（由 Claude 预处理生成 prompt 描述）
- 输出：生成的菜品图片 URL

两个任务并行执行，图片和反馈同时就绪后，整合为结果面板呈现给玩家。

### 数据流（单回合完整流程）

```
阶段一：回合启动
─────────────────────────────────────────────────
前端                      服务端                     外部服务
  │                          │
  ├─ GET /api/round/start ──►│
  │                          ├─ DB 随机查询 ──────────► SQLite
  │                          │◄─ { customer, dish } ───┘
  │                          ├─ 调用 Claude ──────────► Anthropic API
  │                          │◄─ { orderDialogue,        │
  │                          │     ingredients } ────────┘
  │◄─ { customer, dish,      │
  │     orderDialogue,        │
  │     ingredients } ───────┘

阶段二：玩家操作（纯客户端）
─────────────────────────────────────────────────
  玩家选择食材，进行蒸/炒/煎/烤等操作，拼盘
  操作过程记录至 Zustand：operationLog[]

阶段三：提交 & 结果生成（并行）
─────────────────────────────────────────────────
  │                          │
  ├─ POST /api/dish/submit ──►│
  │   { customer, dish,       ├─ 并行触发 ──────────────────────────┐
  │     operationLog }        │                                      │
  │                          ├─ 调用 Claude ──► 食客反馈 (SSE)      │
  │◄─ stream: 反馈文字 ────── │◄─ stream ─────────────────────────  │
  │                          │                                      │
  │                          └─ 调用图像模型 ──────────────────────►│
  │◄─ { imageUrl } ──────────────────────────────────────────────── ┘

结果面板展示：食客反馈文字 + 菜品图片
```

### 提交数据结构（玩家操作 JSON）

```typescript
interface DishSubmission {
  customerId: string;
  dishName: string;
  operationLog: {
    step: number;
    action: 'steam' | 'fry' | 'bake' | 'boil' | 'raw' | 'mix' | 'cut';
    ingredients: string[];
    duration?: string;
    note?: string;
  }[];
  plating: string[];  // 最终拼盘的食材/成品列表
}
```
